The CLI is built to live inside scripts: results go to stdout, errors go to stderr as the JSON-RPC code and message, and exit codes follow one contract — `0` success, `1` command or connection error, `2` usage error. Everything on this page is composition on top of that.

## Machine output

Default output is pretty-printed JSON — pipe it to `jq`:

```bash
godot-mcp engine commands --group node | jq -r '.methods[]'
```

`--format tsv` re-renders a result for classic shell tools: an object becomes `key<TAB>value` rows, an array of objects becomes a header row plus one row per element, and nested values stay as compact JSON. Global flags precede the group:

```bash
godot-mcp --format tsv project info | awk -F'\t' '$1 == "main_scene" { print $2 }'
```

## Drive every node of a type

Queries compose with actions — `batch find-nodes-by-type` returns paths that feed straight back into `node set`:

```bash
godot-mcp batch find-nodes-by-type --type Label \
  | jq -r '.matches[].path' \
  | while read -r p; do
      godot-mcp node set --node-path "$p" --property visible --value false
    done
```

## Preflight, then launch

`godot-mcp doctor` checks the environment (godot binary, project, addon install state, port config, dotnet) and exits `1` only on a real failure — "no editor running yet" is a warning, so it is safe to run before you have launched anything. `godot-mcp status` gives the machine-readable liveness verdict (`running` / `starting` / `crashed` / `closed`); the launch policy follows from it — never start a second editor when one is `running`.

## A complete CI smoke test

```bash
#!/usr/bin/env bash
set -euo pipefail

godot-mcp doctor --project . --json > /dev/null      # environment sanity: exit 1 on real problems

godot --path . --editor --headless &> editor.log &   # one editor, headless
EDITOR_PID=$!
for _ in $(seq 60); do                               # the addon writes the discovery file on bind
  [ -f .godot/godot-mcp.json ] && break; sleep 0.5
done

godot-mcp project info > /dev/null                   # the editor answers
godot-mcp scene play --mode main
godot-mcp runtime eval --code 'emit(get_tree().current_scene.name)' \
  | jq -e '.output[0] == "Main"'                     # assert against the running game
godot-mcp scene stop

kill $EDITOR_PID
```

Every step exits non-zero on failure, so `set -e` turns any broken link into a red build.

## The catalog is data

`engine commands` returns the whole tool surface as JSON — the method list, a group map, and per-command param docs (attached for `--group`, or `--docs` for the full catalog). Generators become trivial: bindings, wrappers, forms, docs that cannot drift.

```bash
# every node command with its required flags
godot-mcp engine commands --group node \
  | jq -r '.docs | to_entries[] | .key + "  " + ([.value.params[]? | select(.required) | "--" + .name] | join(" "))'
```

## Standalone game rigs

With the direct-to-player channel enabled, a QA rig needs no editor at all: run a debug build, then `godot-mcp --game runtime tree`, `--game runtime eval …`, and `--game input …` drive it directly. See [Playtest loop](/docs/playtest-loop#direct-to-player-channel).

<Callout type="tip">
  Running several projects at once? Pin each project's port in its Project Settings (see [Installation](/docs/installation)) — every CLI call then routes itself by the project directory it runs in.
</Callout>
